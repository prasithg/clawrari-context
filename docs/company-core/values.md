# Values & Principles

## Core Principles

### 1. Dogfood Everything
We don't ship patterns we don't use daily. Every skill, every cron, every template runs in our own setup first. If it doesn't survive real production use, it doesn't ship.

### 2. Open Process, Not Just Open Code
The project context repo, FRDs, work orders, decision logs — all public. People learn from seeing how decisions get made, not just the final output. The methodology is as valuable as the code.

### 3. Opinionated Defaults, Escape Hatches Everywhere
Strong defaults that work out of the box. Everything overridable for power users. We'd rather ship 10 strong opinions than 100 "it depends" options.

### 4. Self-Improvement Is The Core Feature
A Clawrari install should measurably improve every week without manual intervention. If it plateaus, that's a bug. The regressions pipeline, RLHF loops, and autoresearch aren't nice-to-haves — they're the product.

### 5. Content Is A First-Class Output
Every feature shipped, every bug fixed, every pattern discovered = potential content. The personal brand flywheel runs on the same fuel as the product. Building in public isn't a marketing strategy — it's the operating model.

### 6. Errors Are Bugs
Cron errors, skill failures, wrong paths, stale configs — these aren't cosmetic. They're bugs. Fix immediately or escalate. A Clawrari setup should run clean.

### 7. Agent-First Engineering
Default to treating problems as code problems. Spawn coding agents for meaty work. Think and reason in-session for questions and research. Build with agents. The human orchestrates — agents execute.

## What We Optimize For

**In order:**
1. Reliability (it works every time)
2. Speed (setup is fast, daily operation is fast, feedback loops are fast)
3. Taste (strong opinions, clean architecture, good defaults)
4. Breadth (covers work + personal + self-improvement)
5. Community (contributions welcome, methodology shared)
